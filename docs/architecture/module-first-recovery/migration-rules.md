# Migration Rules

Use this module-first set as the active normative source.

## Rules for future waves

- start from the scenario, not from a package name
- improve ownership and shorten the reading path in the same wave
- remove, demote, or deprecate at least one old path in every meaningful wave
- add a guardrail for each new hard rule
- do not spend disproportionate effort polishing reserve Telegram capability ahead of active read/reminder/publication concerns

## First execution wave

The first wave after canon replacement is a delta audit.

It must:
- compare current code and docs against this canon
- record what is already true
- record where fake modularity or readability issues remain
- define the next code wave from that evidence instead of continuing refactor inertia
